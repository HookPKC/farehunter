# FareHunter 開發 Playbook（完整版）

> 依據 2026-06 ~ 2026-07 實際開發與踩坑整理。目的：留下可執行的規則，
> 讓任何人（或任何 AI 工具）接手時不重蹈覆轍。短版見 `PLAYBOOK_SHORT.md`，
> 可複製的 prompt 模板見 `PROMPT_TEMPLATES.md`。

## 1. 最容易出錯的 6 類問題（含真實案例）

### 1-1 「部署成功」≠「線上是新版」
- **問題**：repo 有新 code、deploy 顯示 success，但 Pages 實際 serve 舊 commit。
- **真實案例**：航線排序 commit push 後 2 秒即 dispatch deploy，GitHub 解析
  `main` 時仍指向前一個 commit，部署了舊版；驗收時只看了 run success 與
  repo raw 檔，誤報「已上線」。使用者以無痕視窗抓到順序未變才發現。
- **為何像是對的**：run 顯示 success（但建的是舊 SHA）；repo 確實有新 code
  （但那不是部署產物）；前一個 commit 的部分新功能有上線，畫面「看起來有更新」。
- **下次怎麼避免**：部署後核對 deploy run 的 `head_sha == main HEAD`，再查
  `deployments?environment=github-pages` 的 sha；push 與 dispatch 之間留
  ≥30 秒緩衝，或改等 workflow_run 自動觸發。
- **禁用做法**：以「run success + repo raw 檔含新 code」當部署驗收。

### 1-2 外部服務（Google）的行為無法在沙箱驗證
- **問題**：「我們送出的格式正確」不等於「Google 收到後行為正確」。
- **真實案例**：深連結 `q=` 加 `for 1 adult economy class`，本機驗證 189 條
  連結格式全對即部署；實測 Google 解析退化（目的地與日期落空），整功能回退。
- **下次怎麼避免**：外部整合改動一律標記「需真機驗收才算完成」，請 Hook 實際
  點一次拿到確認；未確認前不得宣告完成。
- **禁用做法**：往 `q=` 疊加自然語言；以「URL 編碼正確」推論「落地頁正確」。

### 1-3 同一個數值在兩處各算一次，必然分岔
- **問題**：兩段程式各自計算「最便宜／近期／新鮮」，條件差一點畫面就自相矛盾。
- **真實案例**：(a) Hero（latest 查詢無日期上限）與推薦 cheapest（330 天視窗）
  在同一張卡顯示兩個不同的「最便宜」（7,031 vs 6,014）。(b) 中斷輪次留下未提交
  的 `round100/priceText/isCache`，續作時又新增 `RND100/priceDisp/isCacheSrc`，
  Hero 一度引用已刪除的 helper，險釀 runtime error。
- **下次怎麼避免**：改「被多處使用的數值」前先 grep 所有計算點；修法永遠是抽
  單一共用常數/函式（`NEAR_TERM_DAYS`、`isCacheSrc`）。續作第一步必跑
  `git status` + `git diff`。
- **禁用做法**：複製同一段判斷到第二處；不看 diff 就在被中斷的檔案上續改。

### 1-4 資料真實語意被顯示層「說得比實際更準」
- **問題**：欄位沒錯，但文案讓它看起來更即時、更精確、更可信。本專案最大宗信任殺手。
- **真實案例**：快取 duration 是假值照畫；快取估價印到個位數（6,014）而 Google
  實際 7,762；「觀測於 1 分鐘前」是凍結字串放 5 小時仍顯示 1 分鐘前；快取的
  `observed_at` 只是「抓到快取的時間」卻自稱近期觀測；甚至修正過程本身也踩——
  把「資料更新」改名「資料觀測時間」，但該欄位其實是匯出時間。
- **下次怎麼避免**：改時間/價格文案前，先追欄位「寫入點」確認真實語意；google
  觀測與快取抓取分開表述；相對時間改前端即時換算或絕對時間。
- **禁用做法**：憑欄位名稱推斷語意；凍結字串當即時資訊；快取來源用精確數字。

### 1-5 驗證工具自己也會錯
- **問題**：探針/smoke/selector 出錯時，✗ 可能是探針錯（假陰性）、✓ 可能漏掉真錯（假陽性）。
- **真實案例**：探針用錯元素 id（`upd` vs `statusline`、`nav` vs `routenav`）
  兩度誤報 ✗；Node `toLocaleString` 的 U+2009 細空格讓 regex 誤報；smoke 斷言
  鎖舊文案在改版後假 FAIL；部署誤判（1-1）本質也是驗證量錯對象。
- **下次怎麼避免**：探針報 ✗ 先驗探針（印原始字串/charCode/grep 元素 id）；
  文案改動同輪盤點並更新鎖字串的測試。
- **禁用做法**：看到 ✗ 直接改產品碼；憑記憶猜元素 id 寫探針。

### 1-6 GitHub 排程無聲停擺：零 run ≠ failed run
- **問題**：資料停止更新，但 Actions 一片綠、沒有任何 failure——因為排程事件根本
  沒被投遞，run 從未產生。監控「失敗」抓不到「沒發生」。
- **真實案例**（2026-07-06）：live「資料整理於」停在 04:21；monitor 自 7/5 20:20Z
  的 run #30 後連續 16+ 個整點零排程 run，同期每日 fsc-snapshot、每週 gcal-sweep
  也全缺席（＝repo 級投遞問題，非單一 workflow）；手動 dispatch 卻一切正常。所有
  workflow `state=active`、無停用告示、設定期間零變更。GitHub 官方文件明載整點是
  最高負載時段、排程可能被延遲甚至丟棄，且官方已承認排程丟棄量持續成長。
- **根因判斷法（由源頭往下，每層排除再走下一層）**：①`git log` 看最後 `chore:`
  commit 時間 → ②Actions 看「Scheduled run 是否存在」（不是看 run 有沒有失敗）→
  ③workflow `state` 是否 active → ④其他排程 workflow 是否同時缺席 → ⑤手動
  dispatch 是否正常（區分觸發層 vs 執行層）。
- **復原 SOP**：手動 Run workflow 補資料 → Deploy Pages 若暫時性失敗就 Re-run
  （本案 `deploy-pages@v4` 一次性掛掉、artifact 無恙）→ Disable/Enable workflow
  強制重註冊 → 觀察兩個整點仍死，才改 workflow 檔（改 cron 本身也會重註冊）。
- **防復發**：cron 避開整點（本 repo 已改 `7 * * * *`）；後續可考慮外部 watchdog
  （勿與被監控者同失效域）與前端資料新鮮度警示。
- **禁用做法**：只看 Actions 有無紅字就斷定排程健康；把資料停更直覺歸因於快取或
  部署而不先查源頭；用另一個 GitHub 排程 workflow 監控排程本身。

## 2. 「先驗證再實作」規則

**必先提案、等 Hook 確認再改**：
- 改變使用者看到的價格／順序／推薦語意（價格呈現、排序、推薦文案、通知文案）
- Recommendation Engine 權重、視窗、排名規則
- 外部整合（深連結格式、通知管道）
- UI 結構改動（新增區塊、移位置）——提案階段常會修正方向（例：大字維持最低價）

**必真機/真實頁面驗證（本機綠了不算完）**：
- 深連結與任何 Google 落地行為 → Hook 真機點一次
- 手機版排版 → Hook iOS 實機確認（沙箱無法測）
- 部署 → head_sha + deployment 環境 SHA 雙核對

**不能只看 smoke test**：
- 文案改動（斷言可能鎖舊字串）
- 排序改動（需線上實資料全頁渲染、逐項比對實際順序）
- 多來源顯示（google 與 cache 各挑一案例分別驗）

**只能做最小修改**：`gfLink`、通知模板、latest↔intelligence 對齊區。

## 3. 高風險區域清單（沒有明確授權不碰）

| 區域 | 原因 |
|---|---|
| `gfLink` / Google 深連結 | 實測過加參數會解析退化；現行格式是唯一驗證可用的。要控旅客/艙等須另開 `tfs=` 研究任務 |
| 價格語意文案（觀測/快取/約值） | 六處對齊的一致體系（Hero/日期籤/月度/推薦/次要籤/通知），改任一處可能重生矛盾 |
| 通知模板與其測試 | 3 處測試鎖定文案；通知全是快取價，語意平衡脆弱 |
| `export_web` latest 查詢 ↔ `intelligence._SELECT` | 靠相同視窗＋相同去重才「數學上必然相等」；單邊改動 Hero≠cheapest 立刻復發 |
| `render(data)` 流程 | 一次性渲染；重構（如排序切換完整版）有重繪副作用，需獨立輪次 |
| deploy-pages / monitor workflow | 部署互撞史＋競態史；concurrency 與 workflow_run 是調過的平衡 |
| 引擎 `WEIGHTS` 與信心規則 | 「快取永不 High」「Best 可比 cheapest 貴」是刻意設計，牽動所有解釋文案 |

## 4. 低風險高回報任務類型

- 純文案字串替換（REASONS、footer、圖例）——注意同步更新鎖字串的測試
- 對照表擴充（AIRLINES、SOURCE_LABEL）
- 顯示層排序/過濾（render 入口一段穩定排序，不動資料流）
- 一行說明/導讀/免責（`<p>` + 一行 CSS）
- README 已知限制補充（防重蹈；P1-1 回退紀錄即範例）
- `<details>` 可展開區內容（預設收合，錯了不傷主畫面）

共同前提：每項獨立一輪，改完用實資料渲染驗證。

## 5. 標準工作流程（SOP）

1. **續作先對齊現實**：`git status` + `git diff` + `git log -3`；有殘留先處理殘留。
2. **調查只看直接相關檔案**：grep 定位；結論附證據（行號、實資料數字）。
3. **判斷任務級別**：第 2 節清單內 → 先提案；第 4 節清單內 → 可直接做但仍逐項驗證。
4. **實作最小 diff**：一輪一件事；共用邏輯抽常數；動文案先盤點鎖字串的測試。
5. **驗證三層**：pytest＋smoke → 線上實資料全頁渲染比對具體輸出 → 探針報 ✗ 先驗探針。
6. **部署驗收**：push 後留緩衝再 dispatch；核對 run `head_sha == main HEAD` ＋
   deployment 環境 sha；外部服務行為標記待真機驗收，不得自行宣告完成。
7. **回退條件**：真機實測功能退化 → 立即原樣回退，不在原路線修補；回退後留
   程式註解＋README 紀錄。
8. **不改的條件**：資料成熟度問題（Trend/Fastest 累積中）等資料，不改程式；
   高風險區未授權不碰。
9. **回報格式**：改了哪些檔、前後差異、驗證方式與結果、剩餘已知限制（含
   「我無法在此環境驗證的部分」）。

## 6. 哪些改動一定要人工驗證（Hook 親自）

- Google Flights 深連結落地行為（任何 gfLink 相關改動後，點一條實測）
- 手機版排版（iOS Safari 實機）
- LINE 通知實收樣貌（新模板首次觸發後看一則）
- 部署後線上頁（無痕視窗開一次——本專案已靠這招抓到一次假部署）
- 任何「AI 回報已完成」的部署類任務（5 秒動作：無痕開頁）

## 7. 省 token、減少白做工

- 一輪一件事；範圍模糊的輪次（P0 語意修正撞殘留 helper）最耗 token。
- 禁止重分析專案；歷史脈絡靠 journal/transcript。
- 驗證工具建一次用多輪；文案改版先更新其斷言，避免假 FAIL 燒 token 除錯。
- 批次字串替換用一支 python script 一次完成；單點精準修改才用 str_replace。
- 必拆輪次：調查→提案→實作→驗收各自成輪；「邊查邊改邊部署」是部署事故溫床。
- 高成本模型花在根因分析、語意審計、提案設計；純字串替換/對照表擴充屬機械執行。

## 8. 換工具/換模型時

- 本文件第 3 節（禁區）與 `PLAYBOOK_SHORT.md` 的 10 條守則直接貼進新工具的第一則 prompt。
- 新工具沒有脈絡，會想「優化」刻意設計（快取永不 High、gfLink 不帶參數）——先給禁區清單。
- 任何工具回報「已部署」，Hook 都做無痕開頁 5 秒驗證。
- 秘鑰紀律：token 用完即刪。

## 9. 十條絕對遵守守則

1. 部署驗收看 head_sha 與 deployment 環境 SHA，不看 success 字樣、不看 repo 檔案。
2. push 之後等 30 秒以上才 dispatch，或讓 workflow_run 自動觸發。
3. `q=` 深連結凍結；要改旅客/艙等，另開 `tfs=` 研究任務。
4. google 觀測價＝精確數字；快取/估價＝「約 NT$X」百位化。全站含通知，無例外。
5. 同一數值只准有一個計算點；要用就 import 共用常數，不准複製邏輯。
6. 續作第一步 `git status`/`git diff`；不清楚現況不動手。
7. 改文案先 grep 測試裡的舊字串，同輪一起更新。
8. 探針報錯先驗探針；改產品碼前先確定壞的是產品碼。
9. 語意類改動（價格/時間/推薦/通知）一律先提案，Hook 確認才實作；外部服務行為
   一律真機驗收後才算完成。
10. 一輪只做一件事；回報必含「剩餘已知限制」與「我無法驗證的部分」，不誇大完成度。
